The Baseline

A standard program runs as a single Process with one isolated memory space and a single execution Thread.

Multi-Threading

Spawns new workers inside the same process. Memory is shared. Very lightweight, but risks data corruption.

Multi-Processing

Spawns completely new programs. Memory is duplicated. Safe from race conditions, but heavy on RAM overhead.

Why is Python Slow?

Interpreted

Python isn't pre-compiled. The interpreter translates and executes bytecode line-by-line at runtime, introducing massive instruction overhead as the CPU constantly routes logic on the fly.

Dynamically Typed

Variables lack fixed memory layouts. For every operation, Python must pause to inspect object types and look up methods, severely bottlenecking tight mathematical loops.

The GIL (Global Interpreter Lock)

This mutex ensures only one native thread executes Python bytecode at a time. While it simplifies memory management, it completely prevents true parallel execution for CPU-bound workloads.

High-Level Objects

Everything is a heavy PyObject C-structure, not just raw bytes. This overhead destroys data locality, leading to severe memory fragmentation and constant CPU cache misses.

void pairwise_dist(double** X, double** D, int n, int dim) {
    for (int i = 0; i < n; ++i) {
        for (int j = 0; j < n; ++j) {
            double d = 0.0;
            for (int k = 0; k < dim; ++k) {

                d += diff * diff;
            }
            D[i][j] = sqrt(d);
        }

    int n, int dim

    for (int i = 0; i < n; ++i) {
        for (int j = 0; j < n; ++j) {
            double d = 0.0;

int main(int argc, char *argv[]) {

    int N = atoi(argv[1]);
